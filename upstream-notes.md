# Upstream Notes

## Daily Summary Scale Mismatch

### Context

During the dashboard investigation, the PSD daily summary and the broadband daily summary appeared to live on noticeably different numeric scales. This note records what we found so we do not have to rediscover it later.

### What The Two Endpoints Actually Represent

- `GET /aggregations/daily-summary` in this API is built from `DailyNoiseAnalysis.create_daily_noise_summary_df()`, which loads PSD-band data and groups it by time-of-day.
- Our wrapper then takes the returned PSD-band table and computes a simple arithmetic mean across the selected dB-valued frequency columns between `band_low` and `band_high`.
- `GET /aggregations/daily-broadband-summary` is built from `DailyNoiseAnalysis.create_broadband_daily_noise()`, which loads the dedicated upstream broadband product and groups it by date.

Those are not the same quantity.

Relevant code:

- API wrapper PSD daily summary logic: `app/services/get_aggregations.py`
- Upstream PSD daily summary source: `/workspaces/orcasound/ambient-sound-analysis/src/orcasound_noise/analysis/daily_noise.py`
- Upstream broadband daily summary source: `/workspaces/orcasound/ambient-sound-analysis/src/orcasound_noise/analysis/daily_noise.py`
- Upstream broadband generation math: `/workspaces/orcasound/ambient-sound-analysis/src/orcasound_noise/pipeline/acoustic_util.py`

### Concrete Comparison

For `orcasound_lab` on `2021-10-31`, a direct package check produced:

- broadband daily average from `create_broadband_daily_noise()`: about `1.40`
- PSD-band arithmetic mean across `63-8000 Hz`: about `-11.45`
- PSD energy-sum experiment over the same bands, converted back to dB: about `5.00`

The important point is not the exact numbers but that all three methods land on materially different scales.

### Main Conclusion

This is mostly a semantic mismatch, not an API transport bug.

The current PSD daily summary is:

- based on PSD/octave-band values rather than the dedicated broadband product
- averaged across dB columns with an arithmetic mean
- grouped by time-of-day rather than by date

The broadband daily summary is:

- based on the package's separate broadband product
- one value per day

Because of that, the charts should not be expected to align numerically or visually.

### Important Detail About Our Wrapper

Our current `daily-summary` endpoint is useful as a "typical daily PSD pattern" view, but it is easy to misread as a broadband summary because:

- the route name is generic
- the UI places it near the broadband daily summary
- the response currently exposes `band_low` and `band_high`, which invites comparison to broadband

The README already notes that this is not a true broadband summary. That distinction should remain explicit anywhere this endpoint is shown.

### What Seems Worth Raising Upstream

There are two upstream questions worth bringing to the `orcasound_noise` maintainers.

#### 1. Clarify comparability between broadband and PSD-band summaries

The package exposes:

- `create_daily_noise_summary_df()` for PSD-band daily patterns
- `create_broadband_daily_noise()` for true broadband daily averages

But it does not appear to expose a helper for "band-limited broadband" or for a physically meaningful reduction from PSD-band tables to a broadband-like summary over a chosen frequency range.

Question for maintainers:

- Is there an intended upstream way to derive a broadband-comparable value from the PSD-band products, or are these products intentionally not comparable on absolute scale?

#### 2. Confirm the broadband generation math is intentional

In `/workspaces/orcasound/ambient-sound-analysis/src/orcasound_noise/pipeline/acoustic_util.py`, broadband is generated by summing `abs(DT[i, :])` across FFT bins, scaling by `delta_f`, averaging over time, and then converting to dB.

That is different from the band-power path used elsewhere in the same file, which works from squared quantities before taking a root. Without asserting this is wrong, it is different enough from the PSD-band path that it is worth asking whether this was intentional and whether the resulting broadband values are expected to be directly comparable with PSD-derived band aggregates.

Question for maintainers:

- Is the broadband product intentionally amplitude-summed rather than power-integrated, and should users expect its absolute scale to differ from PSD-band-derived rollups for that reason?

### What Does Not Look Like An Upstream Bug

The mismatch does not currently look like:

- a bug in FastAPI routing
- a serialization bug in this API wrapper
- a frontend charting bug

The bigger issue is that the compared series encode different quantities.

### Local Recommendation

For this API and dashboard:

- continue treating `daily-summary` as a PSD-pattern endpoint, not a broadband endpoint
- avoid presenting its values as directly comparable to the daily broadband series
- if we want a "selected-band broadband-like" daily summary later, add a new endpoint with explicitly documented math rather than overloading the current one

## PSD Heatmap Latency And Aggregation Strategy

### Observed Window-Switch Timings

During dashboard testing with time-window switching:

- `24h`: PSD heatmap about `30.5s` cold, then about `12.2s` warm
- `7d`: PSD heatmap about `63.3s`
- `30d`: PSD heatmap about `226.1s`
- `6h`: PSD heatmap about `7.2s`
- `1h`: PSD heatmap about `6.4s`

By contrast, the API's `daily-summary` endpoint stayed roughly flat around `25s` on cold loads for `1d`, `7d`, and `30d`, then dropped to milliseconds on repeat because of the API cache.

### What The PSD Timings Suggest

The PSD heatmap path appears to be dominated by upstream/raw-read cost rather than by the final number of displayed buckets.

Evidence:

- the `7d` and `30d` PSD responses return similar final bucket counts in the API log (`649` vs `687`)
- despite that, runtime grows from about `63s` to about `226s`
- that means the expensive part is happening before the final matrix is serialized for Plotly

The most plausible contributors, in order, are:

1. upstream `NoiseAccessor.create_df()` work and parquet/S3 reads for each daily chunk
2. API-side pandas resampling and chunk merge
3. JSON serialization and network transfer
4. browser-side Plotly rendering

The current evidence does not support blaming the browser chart as the primary cause of the minute-scale delays.

### Why API-Side Aggregation Still Looks Necessary

There was a question about whether the API is needlessly duplicating Plotly's own aggregation features by aggregating PSD server-side before rendering.

Current conclusion: no, the API-side aggregation still appears justified.

Reasons:

- the dashboard renders a `heatmap` trace from a precomputed `x`, `y`, `z` matrix; it does not hand Plotly a raw per-second point stream plus an instruction to resample
- even if client-side aggregation were possible, sending the full per-second PSD matrix for long windows would dramatically increase payload size and browser memory pressure before any aggregation happens
- the main latency problem already appears to happen upstream of Plotly, during raw PSD loading

### What Plotly Seems To Support Today

Official Plotly.js docs still document `transforms` with `aggregate`, but they explicitly mark transforms as deprecated and say they will be removed in a future version. Plotly.js version 3 also documents removal of previously deprecated features, and the current heatmap docs describe Plotly as rendering a matrix that the application already supplies.

That makes Plotly a poor foundation for core data-reduction logic in this app.

In practice:

- Plotly remains a rendering library here
- the API still needs to decide how much data the browser should receive
- server-side aggregation is the more stable and controllable place to enforce that limit

### Maintainer-Facing Question

If we raise this upstream, the useful question is not "should Plotly aggregate this instead?" The better question is:

- is there a more efficient upstream way to load already-coarsened PSD data for longer windows, so the API does not need to read nearly full-resolution daily PSD chunks just to downsample them immediately?

That is where the largest performance win is most likely to exist.

### Current Answer From The Upstream Repo

From the current upstream implementation, the answer appears to be:

- yes in principle, but not with the current `orcasound_lab` archive inventory
- and not through any existing accessor behavior that automatically derives a coarse PSD read from 1-second PSD files

Why:

- `NoiseAccessor.create_df()` selects archive files by exact `delta_t` and exact frequency-band suffix, downloads those files, and concatenates them
- `S3FileConnector.get_files()` also filters by exact filename suffix such as `1s_3oct.parquet`
- for `orcasound_lab`, the current discovered PSD options show only one archived octave-band resolution: `3oct @ 1s`

So the package can only perform a truly efficient coarse PSD read today if one of these becomes true:

1. the archive contains additional precomputed coarse PSD products such as `3oct @ 60s`, `3oct @ 900s`, or similar
2. the package grows a new accessor path that can stream-resample or lazily aggregate finer-resolution parquet data without materializing nearly the full raw range first

At the moment, neither of those exists in the upstream code path we inspected.
